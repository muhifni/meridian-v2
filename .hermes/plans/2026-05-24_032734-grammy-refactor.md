# Refactor Plan: Migrate Meridian Telegram to grammY

## Goal
Refactor Meridian's Telegram layer dari custom polling + monolithic `index.js` (2200+ lines) ke grammY framework. Hasilnya: modular, maintainable, extensible — tanpa mengubah core logic (agent, screening, management, lessons).

## Why grammY over Telegraf
- Actively maintained (weekly commits 2025-2026, Bot API v9.x support)
- Same middleware architecture as Telegraf (spiritual successor)
- Menu plugin (`@grammyjs/menu`) — declarative inline keyboards, submenus, dynamic buttons, auto-answer
- Conversations plugin (`@grammyjs/conversations`) — plain async/await multi-step flows (better than Telegraf Scenes)
- Router plugin (`@grammyjs/router`) — state-based routing
- TypeScript-first, excellent types
- Migration from raw Telegram API → grammY well-documented
- Active Discord community, responsive maintainer

## Current State
- `telegram.js` (494 lines) — low-level wrapper: sendMessage, editMessage, polling loop, notification helpers
- `index.js` (2247 lines) — EVERYTHING else: command routing (20+ commands), callback query dispatch (30+ inline buttons), cron orchestration, screening/management cycles, menu system, settings UI
- Polling: custom `getUpdates` loop di `telegram.js:363-421`
- Auth: manual `isAuthorizedIncomingMessage()` check
- Command routing: giant if/else chain (~400 lines)
- Callback routing: regex matching di satu function (~200 lines)
- No session, no middleware, no scene
- `setMyCommands` not called — command popup requires BotFather manual setup

## Migration Strategy: Incremental (5 Phases)

### Phase 1 — Install + Bot Instance + setMyCommands (30 min)
**Goal:** grammY running, command popup auto-sync, zero behavior change yet.

Steps:
1. `npm install grammy @grammyjs/menu @grammyjs/conversations @grammyjs/router`
2. Create `telegram/bot.js`:
   ```js
   import { Bot, session } from "grammy";
   const bot = new Bot(process.env.TELEGRAM_BOT_TOKEN);

   // Auto-sync command popup on boot
   await bot.api.setMyCommands([
     { command: "positions", description: "List open positions" },
     { command: "health", description: "Check all external services" },
     { command: "config", description: "Show runtime config" },
     { command: "screen", description: "Refresh candidate list" },
     { command: "menu", description: "Inline command menu" },
     { command: "briefing", description: "Morning briefing" },
     { command: "sim", description: "Dry run stats" },
     { command: "hive", description: "HiveMind sync status" },
     { command: "wallet", description: "Wallet balance" },
     { command: "help", description: "Show all commands" },
   ]);

   export { bot };
   ```
3. Wire `bot.start()` in `index.js` (replace `startPolling()` call)
4. Register one test command: `/help`
5. Verify: bot responds, command popup appears in Telegram

Files changed: `package.json`, new `telegram/bot.js`, `index.js` (swap polling start)
Rollback: revert commit, `npm uninstall grammy`

### Phase 2 — Middleware + Auth (30 min)
**Goal:** Auth, logging, error handling via middleware stack.

Steps:
1. Create `telegram/middleware/auth.js`:
   ```js
   import { log } from "../../logger.js";
   const ALLOWED = new Set(
     String(process.env.TELEGRAM_ALLOWED_USER_IDS || "").split(",").filter(Boolean)
   );
   export function ownerOnly(ctx, next) {
     const userId = String(ctx.from?.id || "");
     if (ALLOWED.size > 0 && !ALLOWED.has(userId)) return; // silent drop
     return next();
   }
   ```
2. Create `telegram/middleware/logging.js`:
   ```js
   import { log } from "../../logger.js";
   export function logUpdates(ctx, next) {
     const text = ctx.message?.text || ctx.callbackQuery?.data || "";
     log("telegram", `${ctx.updateType}: ${text}`);
     return next();
   }
   ```
3. Create `telegram/middleware/error.js`:
   ```js
   import { log } from "../../logger.js";
   export function setupErrorHandler(bot) {
     bot.catch((err) => {
       log("telegram_error", `${err.ctx?.updateType || "unknown"}: ${err.message}`);
     });
   }
   ```
4. Register in `bot.js`:
   ```js
   bot.use(ownerOnly);
   bot.use(logUpdates);
   setupErrorHandler(bot);
   ```
5. Test: send message from unauthorized user → no response. Send from King → logged + processed.

Files changed: new `telegram/middleware/*.js`, `telegram/bot.js`

### Phase 3 — Command Handlers + Menu Plugin (2-3 hours)
**Goal:** Extract all commands from `index.js` if/else chain. Replace manual inline keyboards with Menu plugin.

Steps per command group:
1. Create handler file (e.g., `telegram/commands/health.js`):
   ```js
   import { Composer } from "grammy";
   export const healthCmd = new Composer();
   healthCmd.command("health", async (ctx) => {
     await ctx.reply("🔍 Running health checks...");
     const results = await runAllChecks();
     await ctx.reply(formatResults(results));
   });
   ```
2. Import and register in `bot.js`: `bot.use(healthCmd);`
3. Remove corresponding if/else from `index.js`
4. Test each command after migration

**Menu plugin — replace manual inline keyboards:**
```js
import { Menu, MenuRange } from "@grammyjs/menu";

const mainMenu = new Menu("main")
  .text("📊 Positions", (ctx) => showPositions(ctx)).row()
  .text("⚙️ Settings", (ctx) => ctx.menu.nav("settings")).row()
  .text("🔍 Screening", (ctx) => showCandidates(ctx)).row()
  .text("📈 Briefing", (ctx) => showBriefing(ctx)).row()
  .text("🏥 Health", (ctx) => runHealthCheck(ctx));

const settingsMenu = new Menu("settings")
  .dynamic(async (ctx, range) => {
    const cfg = loadConfig();
    range.text(`Strategy: ${cfg.strategy}`, (ctx) => toggleStrategy(ctx)).row();
    range.text(`Max Positions: ${cfg.maxPositions}`, (ctx) => cycleMaxPos(ctx)).row();
    range.text(`Trailing TP: ${cfg.trailingTp}%`, (ctx) => cycleTrailingTp(ctx)).row();
    range.text(`Min Organic: ${cfg.minOrganic}`, (ctx) => cycleMinOrganic(ctx)).row();
  })
  .back("← Back to Menu");

mainMenu.register(settingsMenu);
bot.use(mainMenu);

bot.command("menu", (ctx) => ctx.reply("🎛 Meridian Control Panel", { reply_markup: mainMenu }));
```

Benefits vs current approach:
- Auto-answer callback queries (no more manual `answerCallbackQuery`)
- Submenu navigation built-in (`ctx.menu.nav("settings")`)
- Dynamic buttons regenerate on each render (always shows current config values)
- No more giant callback dispatch function

   async function deployWizard(conversation, ctx) {
     await ctx.reply("Pilih pool (ketik address atau index dari /candidates):");
     const { message } = await conversation.wait();
     const pool = message.text;

## Notification Handling (Cron-triggered, no ctx)
Cron cycles (screening, management) send notifications without incoming update context.
Solution: use `bot.api.sendMessage(chatId, text)` directly:
```js
// In screening cycle, management cycle, etc.
import { bot } from "./telegram/bot.js";
const CHAT_ID = process.env.TELEGRAM_CHAT_ID;
await bot.api.sendMessage(CHAT_ID, "🧪 Deployed DEGEN-SOL (DRY RUN)");
```
This replaces current `sendMessage()` from `telegram.js` — same behavior, just through bot instance.

## Live Message System Adaptation
`createLiveMessage()` currently uses raw `postTelegram()`. Adapt to use `bot.api`:
```js
// Instead of: postTelegram("sendMessage", { text, chat_id })
// Use: bot.api.sendMessage(chatId, text)
// Instead of: postTelegram("editMessageText", { text, message_id, chat_id })
// Use: bot.api.editMessageText(chatId, messageId, text)
```
Minimal change — same logic, different transport.

## Risks & Tradeoffs
1. **Breaking existing behavior** — incremental migration reduces risk. Each phase independently testable. Rollback = revert git commit.
2. **Session state on restart** — in-memory session lost on PM2 restart. Mitigation: use file-based adapter (`@grammyjs/storage-file`) or keep critical state in existing JSON files (state.json, lessons.json).
3. **Live message system** — `createLiveMessage()` tightly coupled to current raw fetch. Needs adapter to `bot.api`. Low risk — same API calls, different wrapper.
4. **Cron-triggered messages** — cron cycles send notifications without `ctx`. Solution: `bot.api.sendMessage(chatId, ...)` — documented pattern in grammY.
5. **Menu plugin learning curve** — new concept (declarative menus). But eliminates 200+ lines of manual callback dispatch. Net positive.
6. **Conversations plugin state** — conversation state lives in memory by default. If bot restarts mid-conversation, wizard resets. Mitigation: conversations are short (2-3 steps), restart is acceptable.
